Introduction

Développement et déploiement d'un micro-service permettant de générer des PDF.
Le micro-service sera appelé par un nouveau projet Symfony.
Le projet Symfony devra avoir une interface utilisateur ( codée en Twig ) permettant, dans un premier temps, la création de compte, la connexion et la possibilité de changer son niveau d'abonnement.
En fonction du niveau d'abonnement, l'utilisateur aura accès à plus ou moins de fonctionnalité ( ex : 2 pdfs par jour vs illimité ).
Les grands objectifs de ce module sont :

  • Création d'un micro-service de génération de PDF :

    • Comprendre les concepts fondamentaux des micro-services.
    • Développer un micro-service spécifique pour la génération de fichiers PDF (utilisation de gotenberg).
    • Suivre les bonnes pratiques de conception et de développement pour les micro-services.
    • Utilisation de Symfony HttpClient

  • Intégration avec un projet Symfony :

    • Créer un nouveau projet Symfony qui utilisera notre micro-service pour générer des PDF.
    • Configurer l'intégration entre le micro-service et le projet Symfony.
    • Utilisation de Symfony HttpClient

     

  • Interface Utilisateur avec Twig :

    • Concevoir et développer une interface utilisateur avec Twig, le moteur de templates de Symfony.
    • Mise en place de fonctionnalités telles que la création de compte et la connexion.
    • Gérer les différents niveaux d'abonnement de l'utilisateur.

  • Gestion des niveaux d'abonnement :

    • Mettre en place un système permettant aux utilisateurs de choisir leur niveau d'abonnement.
    • Limiter l'accès à certaines fonctionnalités en fonction du niveau d'abonnement sélectionné.
    • Implémenter des mécanismes pour permettre aux utilisateurs de changer leur niveau d'abonnement.

  • Déploiements auto.

    • Installation et déploiement de gotenberg
    • Installation et déploiement de Symfony

4 - Les micro-services

Un micro-service est une architecture logicielle qui organise une application sous forme d'un ensemble de services autonomes, indépendants, et spécialisés.

Chaque micro-service est une unité qui accomplit une tâche spécifique dans un contexte global.

Ils sont conçus pour être déployés, développés et évolués indépendamment les uns des autres. 

  1. Indépendance : Chaque micro-service est indépendant, ce qui signifie qu'il peut être développé, déployé et mis à l'échelle de manière autonome. Cela facilite la maintenance, la mise à jour et l'évolutivité.

  2. Autonomie : Chaque micro-service dispose de son propre code, base de données et dépendances. Il peut être développé dans différents langages (PHP, Python, etc) et peut même être déployé sur plusieurs plates-formes.

  3. Communication via API : Les micro-services interagissent entre eux via API.

  4. Déploiement et scalabilité : Les micro-services peuvent être déployés séparément.
    Chaque micro-service peut être mis à l'échelle indépendamment en fonction de la charge qu'il reçoit.

  5. Maintenance : La maintenance est simplifiée, car chaque service est autonome. Les mises à jour ou les correctifs peuvent être appliqués à un micro-service sans affecter l'ensemble de l'application.

  6. Gestion des données : Chaque micro-service peut avoir sa propre base de données ou partager des bases de données avec d'autres services.

  7. Résilience : En cas de panne d'un micro-service, les autres services continuent de fonctionner normalement.

4.1 - HTTPClient

Symfony communique avec un micro-service via Symfony en utilisant des requêtes HTTP ( module HTTPClient ).

Vous devez vous assurer que Symfony HttpClient est installé dans votre projet Symfony

composer require symfony/http-client

Utilisation de Symfony HttpClient pour les requêtes HTTP : 

Doc officielle : https://symfony.com/doc/7.3/http_client.html

Injection de dépendances : Pour utiliser Symfony HttpClient dans vos contrôleurs, services ou autres composants Symfony, vous devez l'injecter en tant que dépendance.

use Symfony\Contracts\HttpClient\HttpClientInterface;

Effectuer une requête : Une fois le client HttpClient injecté, vous pouvez l'utiliser pour effectuer des requêtes HTTP vers votre micro-service.

use Symfony\Contracts\HttpClient\HttpClientInterface;

class SymfonyDocs
{
    public function __construct(
        private HttpClientInterface $client,
    ) {
    }

    public function fetchGitHubInformation(): array
    {
        $response = $this->client->request(
            'GET',
            'https://api.github.com/repos/symfony/symfony-docs'
        );

        $statusCode = $response->getStatusCode();
        // $statusCode = 200
      
        $contentType = $response->getHeaders()['content-type'][0];
        // $contentType = 'application/json'
        
	    $content = $response->getContent();
        // $content = '{"id":521583, "name":"symfony-docs", ...}'
        
      	$content = $response->toArray();
        // $content = ['id' => 521583, 'name' => 'symfony-docs', ...]

        return $content;
    }
}

Options de requête : Vous pouvez également spécifier diverses options de requête telles que les en-têtes, les paramètres de requête, le contenu de la requête, etc.

// Exemple avec des options de requête
$response = $this->client->request('POST', 'https://example.com/api/post', [
    'headers' => [
        'Content-Type' => 'application/json',
    ],
    'body' => json_encode(['key' => 'value']),
]);

6 - Formulaire Twig

Créez un formulaire Twig permettant au client de saisir une URL afin de générer un PDF

composer require symfony/form

Exemple de Controller : 

// src/Controller/GeneratePdfController.php

namespace App\Controller;

use App\Service\YourGotenbergService; // Importez votre service ici
use Symfony\Bundle\FrameworkBundle\Controller\AbstractController;
use Symfony\Component\HttpFoundation\Request;
use Symfony\Component\HttpFoundation\Response;

class GeneratePdfController extends AbstractController
{
    private $pdfService;

    public function __construct(YourGotenbergService $pdfService)
    {
        $this->pdfService = $pdfService;
    }

    public function generatePdf(Request $request): Response
    {
        // Créer le formulaire
        $form = $this->createFormBuilder()
            ->add('url', null, ['required' => true])
            ->getForm();

        // Gérer la soumission du formulaire
        $form->handleRequest($request);

        // Si le formulaire est soumis et valide
        if ($form->isSubmitted() && $form->isValid()) {
            // Récupérer l'URL saisie à partir des données du formulaire
            $url = $form->getData()['url'];

            // Faites appel à votre service pour générer le PDF
            $pdf = $this->pdfService->generatePdfFromUrl($url);

            // Rediriger ou afficher une réponse appropriée
            // Par exemple, rediriger vers une page de confirmation
            return $this->redirectToRoute('pdf_generated_success');
        }

        // Afficher le formulaire
        return $this->render('pdf/generate_pdf.html.twig', [
            'form' => $form->createView(),
        ]);
    }
}

Exemple de Twig :

{# templates/pdf/generate_pdf.html.twig #}

{% extends 'base.html.twig' %}

{% block title %}Generate PDF{% endblock %}

{% block body %}
    <h1>Generate PDF</h1>

    <form method="post" action="{{ path('pdf_generate') }}">
        <label for="url">URL:</label>
        <input type="text" id="url" name="url" required />
        <button type="submit">Generate PDF</button>
    </form>
{% endblock %}

ATTENTION : ux-turbo est activé par défaut, penser à le désactiver si vous n'en avez pas l'utilité, il peut provoquer des effets de bord sur votre développement actuel.

composer remove symfony/ux-turbo

Ou rajouter l'attribut data-turbo="false" à votre formulaire pour qu'il n'utilise pas turbo : 

{{ 
    form_start(form, {
        'attr': { 'data-turbo': "false" }
    }) 
}}

// src/Controller/SecurityController.php

namespace App\Controller;

use Symfony\Bundle\FrameworkBundle\Controller\AbstractController;
use Symfony\Component\HttpFoundation\Request;
use Symfony\Component\HttpFoundation\Response;
use Symfony\Component\Security\Http\Authentication\AuthenticationUtils;

class SecurityController extends AbstractController
{
    public function login(AuthenticationUtils $authenticationUtils): Response
    {
        // get the login error if there is one
        $error = $authenticationUtils->getLastAuthenticationError();

        // last username entered by the user
        $lastUsername = $authenticationUtils->getLastUsername();

        return $this->render('security/login.html.twig', ['last_username' => $lastUsername, 'error' => $error]);
    }
}

9 - Custom query on repository

Vous avez besoin de connaitre le nombre de PDF générer sur un jour donné par utilisateur.

Cette requête doit vous permettre de bloquer la génération de PDF dés que la limite donnée par l'abonnement est atteinte.

Ajout d'une fonction dans le repository de votre entity Pdf

La méthode getSingleScalarResult() est utilisée pour exécuter une requête qui retourne un seul résultat sous forme scalaire (c'est-à-dire un seul champ / une seule valeur)

Dans le cas où vous utilisez COUNT(p.id) dans votre requête, cela retournera un seul nombre représentant le nombre de résultats trouvés.

<?php

namespace App\Repository;

use App\Entity\Pdf;
use Doctrine\Bundle\DoctrineBundle\Repository\ServiceEntityRepository;
use Doctrine\Persistence\ManagerRegistry;

/**
 * @extends ServiceEntityRepository<Pdf>
 *
 * @method Pdf|null find($id, $lockMode = null, $lockVersion = null)
 * @method Pdf|null findOneBy(array $criteria, array $orderBy = null)
 * @method Pdf[]    findAll()
 * @method Pdf[]    findBy(array $criteria, array $orderBy = null, $limit = null, $offset = null)
 */
class PdfRepository extends ServiceEntityRepository
{
    public function __construct(ManagerRegistry $registry)
    {
        parent::__construct($registry, Pdf::class);
    }

  public function countPdfGeneratedByUserOnDate($userId, $startOfDay, $endOfDay)
  {

      return $this->createQueryBuilder('p')
          ->select('COUNT(p.id)')
          ->where('p.user = :userId')
          ->andWhere('p.createdAt BETWEEN :startOfDay AND :endOfDay')
          ->setParameter('userId', $userId)
          ->setParameter('startOfDay', $startOfDay)
          ->setParameter('endOfDay', $endOfDay)
          ->getQuery()
          ->getSingleScalarResult();
  }

10 - Cypress

Sources : 

  • https://cypress.io/
  • Doc Cypress

Cypress est un framework de tests end to end open source conçu pour la mise en plce de tests automatisés pour les applications web.

  1. Tests de bout en bout (E2E) : Cypress permet de réaliser des tests end to end, ce qui signifie qu'il simule les actions d'un utilisateur réel sur une application web, telles que la navigation, la saisie de données et l'interaction avec les éléments de l'interface utilisateur.

  2. Architecture orientée client : Cypress est construit sur une architecture orientée client, ce qui signifie qu'il s'exécute dans le même environnement que votre application web. Cela lui permet d'accéder directement au DOM, aux cookies, aux variables JavaScript et aux API du navigateur, ce qui facilite la simulation des actions utilisateur.

  3. UI : Cypress propose une interface utilisateur riche qui affiche les résultats des tests, les captures d'écran, les vidéos de test et les erreurs de manière claire et intuitive. Vous pouvez également filtrer, trier et rechercher les tests pour une meilleure gestion.

  4. Intégration de Cypress à n’importe quelle CI : exemple de Github Actions


Installer Cypress ( dans le projet Symfony ) :

npm install cypress --save-dev

Création des tests Cypress :

Dans votre projet, créez un nouveau dossier appelé cypress à la racine du projet. À l'intérieur du dossier cypress, créez un dossier appelé e2e. C'est ici que vos fichiers de test seront situés.

Dans le dossier e2e, créez un nouveau fichier de test, par exemple login.cy.js qui contiendra votre premier test Cypress pour le formulaire de connexion.

Premier test :

describe('Formulaire de Connexion', () => {
  it('test 1 - connexion OK', () => {
    cy.visit('YOU_LOCAL/login');

    // Entrer le nom d'utilisateur et le mot de passe
    cy.get('#username').type('YOUR_USERNAME');
    cy.get('#password').type('YOUR_PASSWORD');

    // Soumettre le formulaire
    cy.get('button[type="submit"]').click();

    // Vérifier que l'utilisateur est connecté
    cy.contains('You are logged in as YOUR_USERNAME').should('exist');
  });

  it('test 2 - connexion KO', () => {
    cy.visit('YOUR_LOCAL/login');

    // Entrer un nom d'utilisateur et un mot de passe incorrects
    cy.get('#username').type('YOUR_USERNAME');
    cy.get('#password').type('WRONG_PASSWORD');

    // Soumettre le formulaire
    cy.get('button[type="submit"]').click();

    // Vérifier que le message d'erreur est affiché
    cy.contains('Invalid credentials.').should('exist');
  });
});

Lifecycle callbacks are defined as public methods inside the entity you want to modify. For example, suppose you want to set a createdAt date column to the current date, but only when the entity is first persisted (i.e. inserted). To do so, define a callback for the prePersist Doctrine event:

C'est à dire : 

Les Lifecycle callbacks sont définis en tant que méthodes publiques au sein de l'entité que vous souhaitez modifier.

Par exemple, si vous souhaitez définir une colonne createdAt à la date actuelle :

// src/Entity/YourEntity.php
namespace App\Entity;

use Doctrine\ORM\Mapping as ORM;

#[ORM\Entity]
#[ORM\HasLifecycleCallbacks]
class YourEntity
{
    // ...

    #[ORM\PrePersist]
    public function setCreatedAtValue(): void
    {
        $this->createdAt = new \DateTimeImmutable();
    }
}

Les 2 points importants sont : 

#[ORM\HasLifecycleCallbacks]

Et :

    #[ORM\PrePersist]
    public function setCreatedAtValue(): void
    {
        $this->createdAt = new \DateTimeImmutable();
    }

Twig est un moteur de template pour PHP qui facilite la création de vues dans les applications web.

https://twig.symfony.com/doc/

Variables : Vous pouvez utiliser des variables dans vos templates Twig pour afficher des données dynamiques. Par exemple, {{ user.username }} afficherait le nom d'utilisateur actuellement connecté.

Structures de contrôle : Twig prend en charge des structures de contrôle telles que les boucles ({% for %}), les conditions ({% if %}), les blocs ({% block %}), les includes ({% include %}), etc. 

Filtres : Vous pouvez utiliser des filtres pour modifier les variables avant de les afficher.
Par exemple, {{ user.username|capitalize }} capitaliserait le nom d'utilisateur.
